Bitcoinの拡張秘密鍵を保存するのに、暗号化してwincodeでシリアライズして保存する仕組みを作っていた。
[こんなの](https://github.com/hirokuma/rust-wallet-utils/blob/62d634d587cb7b7b04a53ec098e0e37f3f50e421/src/encdec.rs#L71-L76)だ。
別に拡張秘密鍵だけでなくてもよいので、文字列全般にしている。

```rust
#[derive(wincode::SchemaWrite, wincode::SchemaRead, Deserialize, Debug)]
struct FormatV1<'a> {
    salt_bytes: &'a [u8],
    nonce_bytes: &'a [u8],
    ciphertext: &'a [u8],
}
```

わざわざlifetimeを使っているものの、大きなデータにならないし扱いが面倒だから`&`を外してみた。
saltとnonceは固定長、暗号化したデータは`Vec<u8>`で可変長。

```rust
#[derive(wincode::SchemaWrite, wincode::SchemaRead, Deserialize, Debug)]
pub struct FormatV1 {
    pub salt_bytes: [u8; SALT_LEN_V1],
    pub nonce_bytes: [u8; NONCE_LEN_V1],
    pub ciphertext: Vec<u8>,
}
```

これは内部で元々使っているデータだったので、その定義をstructにしただけのようなものだ。  
なので変更もすんなり行くだろうと思っていた。

保存したデータを読み込むのはできると思うので、まだ使っている以前に保存していたファイルを読み込むだけの実験を先に行った。  
これが`wincode::deserialize()`で`PreallocationSizeLimit`エラーになるのだ。

どうやら、`&[u8]`のようなデータは先にデータ長を保存してデータが続くようになっているが、
`[u8; X]`のように固定長となっている場合はいきなりデータで始まるようになっているらしい(Gemini)。

## 読み取り側の実装に依存

いやいや、サイズがなくてどうやって読み取りサイズを決めるのよ。。。  
と思ったら、そこは`[u8; X]`の定義を信用して読み取るらしい。
エンコードされたデータはファイルフォーマットではなくデータが詰まっているだけで、
読み取り側がどう料理するか決めるということだ。

それが嫌ならデータがフォーマットを持つようにせねばなるまい。  
Geminiは`ciborium`、`rmp-serde`、`postcard`なんかを候補に挙げてくれた。
それぞれ特徴があるようなので必要があれば考えようかね。
