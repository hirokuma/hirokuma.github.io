[前回の最後](https://blog.hirokuma.work/2026/07/20260701-rust.html#thiserror%E3%81%A7%E3%83%AD%E3%82%B0%E3%82%92%E5%87%BA%E3%81%97%E3%81%9F%E3%81%84)で、ログが長いのでうざったいということを書いた。

```log
2026-07-02T14:21:59.033928Z ERROR btc_wallet: [src/lib.rs:215] Backend error occurred(reason=load: ElectrumRpc::new)> Electrum client error occurred: reason=electrum_client::Client::new(tcp://127.0.0.1:50001), source=Connection refused (os error 111)
```

経路がわかるからよいのだが、それにしてもちょっと長過ぎる。

ちょっと[Gemini](https://share.gemini.google/PxZ8e11r4TwY)にご相談。

> thiserror の基本スタンスは、#[error("...")] で書く Display は最外殻のクリーンなメッセージ に留め、source の連鎖（経路）は Debug（あるいは anyhow や eyre のようなエラー報告ツール）に任せる という設計です。

えええ。
無料版Geminiだから適当なことを言ってないだろうか。  
とはいえ、長いんだから短くすればいいというのは当たり前の話だ。  
`source`をエラーログのために使っていたが、`{:?}`を使えば全部表示されるし、
条件分岐したければ直接見ることもできるのでそれでよい気がしてきた。

`{e}`だけだとこのくらいで、

```log
2026-07-03T13:55:02.745135Z ERROR sample: wallet: Backend error occurred: load: ElectrumRpc::new
```

`{e:?}`だとここまで出力される。

```log
2026-07-03T13:53:59.692905Z ERROR sample: wallet: Backend { reason: "load: ElectrumRpc::new", source: Electrum { reason: "electrum_client::Client::new(tcp://127.0.0.1:50001)", source: IOError(Os { code: 111, kind: ConnectionRefused, message: "Connection refused" }) } }
```

もしかしたらこれで`cargo clippy`での"very large" warningが出なくなるかも、と一瞬思ったがそんなことはなかった。
Displayは関係ないしね。

その代わり、`file!()`や`line!()`で今の箇所をログ出力するマクロを作って埋め込んでいった。  
全部の箇所に埋め込んだので大変だったが、このくらいならやるしかあるまい。


```log
2026-07-03T14:33:10.541268Z ERROR btc_wallet::electrum: [src/electrum.rs:28] Electrum client error occurred: electrum_client::Client::new(tcp://127.0.0.1:50001)
2026-07-03T14:33:10.541322Z ERROR btc_wallet: [src/lib.rs:222] Backend error occurred: load: ElectrumRpc::new
2026-07-03T14:33:10.541480Z ERROR sample: wallet: Backend error occurred: load: ElectrumRpc::new
Error: Backend error occurred: load: ElectrumRpc::new

Caused by:
    0: Electrum client error occurred: electrum_client::Client::new(tcp://127.0.0.1:50001)
    1: Connection refused (os error 111)
```

## sourceなしだと分かりづらい

数日置いて冷静になった。

アプリ側に返す`err`としては`source`があるので全部返っているのだが、ログとして出力させると直近しかわからない。
この程度のログの量なら良いのだが、場合によってはログが他と入り乱れることがあり、そうなるとspanで経路がわかるようにするか文字列としてわかるようにするかしておかないとつらいかもしれない。

いや、そうだろうか？  
エラーが起きた箇所では詳細なログがあるのだ。それを全部の経路で表示を引き継ぐ必要があるんだろうか。
`tracing`限定ではあるがspanだけでも十分かもしれない。

軽く見てみよう。  
説明してなかったが、`load()`という関数がさらに下側を呼び出してエラーが起きて戻していって`main()`でもそのまま終わらせたログの例だ。
`load()`には`#[instrument]`をつけている。  
実際に起きたエラーは`Connection refused (os error 111)`。
それを`thiserror`のエラーに`.map_err`で変換し、`load()`でもさらに`thiserror`でくるんで返している。
最後の"Caused by"は`main()`を抜けた後だ。

```log
2026-07-06T12:54:17.757688Z ERROR load: btc_wallet::electrum: src/electrum.rs:28: new instance(Electrum: Connection refused (os error 111)): config.server=tcp://127.0.0.1:50001
2026-07-06T12:54:17.757781Z ERROR load: btc_wallet: src/lib.rs:196: backend: new instance(Electrum: Connection refused (os error 111)): config.server=tcp://127.0.0.1:50001
2026-07-06T12:54:17.758239Z ERROR sample: examples/sample.rs:46: Fail wallet instance creation: backend: new instance(Electrum: Connection refused (os error 111)): config.server=tcp://127.0.0.1:50001
Error: backend: new instance(Electrum: Connection refused (os error 111)): config.server=tcp://127.0.0.1:50001

Caused by:
    0: new instance(Electrum: Connection refused (os error 111)): config.server=tcp://127.0.0.1:50001
    1: Electrum: Connection refused (os error 111)
    2: Connection refused (os error 111)
```

これでクレートのログなしにするとこんな感じになる。  
別にクレートをつくる人でなければログは出さなくてもいいと思うだろうから、これでいいのかな。
