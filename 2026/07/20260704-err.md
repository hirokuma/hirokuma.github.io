タグをRust言語にしたが特にそういう制限はないと思う。  
いつも悩んでいる、ログにはどういう情報を出力したらよいのかについてだ。

## 小さい組み込みソフトウェアの場合

私が一番長いことやっていたのは小さい組み込みソフトウェア環境である。  
基本的にはC言語で、JTAGでログが出せるなら幸いだし、UARTで出すことができてもうれしい。
が、だいたいはJTAGやUARTで出力すると実際にやらんといかん処理のほうが邪魔されるので、
そういうしっかりしたログを出すとしたらハングするルートだけにしておく(そしてハングするのでほぼ出せない)。  
だいたいはLEDの点滅のさせ方だったり、そういうものがなければ空いているGPIOをHI/LOさせてロジアナで見る、みたいなことをしていた。

つまり、ログの文字列を出力するという経験がほとんど無いのだ！  
今はLinuxが使えるなどのリッチな環境で作業することがほとんどなのだが、
どうにもよいログ文字列を考えつくことができない。  
そう、1DKの部屋で生活していた人が2階建ての一軒家に移り住んでもどう空間を使ってよいかわからないように！

なんか力強く言い張れば許されるんじゃないかと思っている。

## Rustのexpect

Rustの`.expect("～")`はこういう文字列にしましょうという指針があったはずだ。
確か、エラーの理由ではなく期待する動作を書く、だったと思う。  
ここか。

* [Recommended Message Style](https://doc.rust-lang.org/std/result/enum.Result.html#recommended-message-style)
  * > 結果が OK であるはずであると期待する理由を説明するために、期待メッセージを使用することをお勧めします。
  * > 予期されるエラー メッセージの表現方法を覚えるのが難しい場合は、「環境変数は何らかの方法で設定する必要があります」または「指定されたバイナリは現在のユーザーが利用可能で実行可能である必要があります」のように「すべき」という単語に注目してください。
  * > 予期されるメッセージ スタイルと推奨の根拠の詳細については、std::error モジュールのドキュメントの「Common Message Styles」のセクションを参照してください。
* [Common Message Styles](https://doc.rust-lang.org/std/error/index.html#common-message-styles)
  * > 人々がメッセージを期待する言い方には 2 つの一般的なスタイルがあります。
  * "expect as error message" (「エラー メッセージとして期待」)
    * メッセージを使用してパニックに遭遇したユーザーに情報を提示
  * "expect as precondition" (「前提条件として期待」)
     メッセージを使用してパニックをデバッグしている開発者に情報を提示

例にあるのは`std::env::var("IMPORTANT_PATH")`に対するこういうメッセージ。

* `IMPORTANT_PATH`変数が設定されていない
* `IMPORTANT_PATH`変数は`wrapper_script.sh`で設定されるべき

前者は「ソースコード見ればわかるよ」ということなんだと思う。
`.expect()`はその場でpanicするからソースコードの位置もわかるのだからそんな情報を出しても無駄だから、
追加できるとしたらここで期待する動作について書くことなのだろう。

## 私のログルール(暫定)

これはpanicしてソースコードの位置がわかることが前提となっているはずだ。  
一般的には、エラーが起きて自分で判断できないのであれば上の方まで伝搬させ、
panicさせずなるべくアプリの上の方でエラーをユーザーに見せて終わらせることが多いのではなかろうか。
`main() -> Result<>`でエラーを返してメッセージはRust側に任せるかもしれないが。

panicしないログはユーザーに見せないものなのでそういう制約は気にせず、
開発者がストレスなく解析するために役立つ情報が載っているのがよいはずだ。
まあ、ストレスなく解析するというのは無理かもしれんが。。。

複数の人で作るときは、先にログをどうするか決めておかないと
実装した人によってログのフォーマットやレベルが違うとイライラするだろう。

「何を当たり前な」と思うかもしれないが、エラー発生＝CPUリセットという世界だとそんなにログについて意識合わせしなかったりするのだ。
システムを作るとき、私はそういうのに極力かかわらずに慣れた人にお任せしたいね。
私にさせてはダメだ。  
今回は、私が一人で実装するときの指針がないので決めているだけだ。

### エラーの箇所をログだけでわかるようにしたい

ソースコードを見ればエラーの種類がわかると言っても、ログファイルを見てファイル名と行番号だけしかなかったら腹が立つだろう。
なので、タグジャンプできるようにファイル名と行番号は出力しつつ、エラーの内容も文字列だけで多少はわかるようにしておきたい。
そして、気になったログだけタグジャンプして見に行くのが理想だ。

場所がわかっても、この値がこうだからエラーだというような情報があれば出力したい。
そうなると`&'static str`よりも`String`になるのか。  
最初の頃はファイル名や行番号の出力ができると思っていなかったのでユニークな文字列を埋め込んでいたので`&'static str`を使っていた。
たとえば`reason`がそうだ。

```rust
    #[error("WinCode read error({source}): {reason}")]
    WinCodeRead {
        reason: &'static str,
        #[source]
        source: wincode::ReadError,
    },
```

C言語では、`__FILE__`と`__LINE__`だけしかなかったのだが、GNU拡張で`__FUNCTION__`があったし C99からだったか`__func__`が追加されて、
現在の関数名もログに出せるようになった。  
Rustで外部クレート無しで関数名を出すのは面倒なようだ。
tracingでも出せるようだが、`#[instrument]`を付けたところだけになるとかなんとか。調査不足だ。

まあ、それならそれでよいか。
