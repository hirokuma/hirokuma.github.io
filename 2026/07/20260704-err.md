タグをRust言語にしたが特にそういう制限はないと思う。  
いつも悩んでいる、ログにはどういう情報を出力したらよいのかについてだ。

## 小さい組み込みソフトウェアの場合

私が一番長いことやっていたのは小さい組み込みソフトウェア環境である。  
基本的にはC言語で、JTAGでログが出せるなら幸いだし、UARTで出すことができてもうれしい。
が、だいたいはJTAGやUARTで出力すると実際にやらんといかん処理のほうが邪魔されるので、
そういうしっかりしたログを出すとしたらハングするルートだけにしておく(そしてハングするのでほぼ出せない)。  
だいたいはLEDの点滅のさせ方だったり、そういうものがなければ空いているGPIOをHI/LOさせてロジアナで見る、みたいなことをしていた。

つまり、ログの文字列を出力するという経験がほとんど無いのだ！  
今はLinuxが使えるなどのリッチな環境で作業することがほとんどなのだが、
どうにもよいログ文字列を考えつくことができない。  
そう、1DKの部屋で生活していた人が2階建ての一軒家に移り住んでもどう空間を使ってよいかわからないように！

なんか力強く言い張れば許されるんじゃないかと思っている。

## Rustのexpect

Rustの`.expect("～")`はこういう文字列にしましょうという指針があったはずだ。
確か、エラーの理由ではなく期待する動作を書く、だったと思う。  
ここか。

* [Recommended Message Style](https://doc.rust-lang.org/std/result/enum.Result.html#recommended-message-style)
  * > 結果が OK であるはずであると期待する理由を説明するために、期待メッセージを使用することをお勧めします。
  * > 予期されるエラー メッセージの表現方法を覚えるのが難しい場合は、「環境変数は何らかの方法で設定する必要があります」または「指定されたバイナリは現在のユーザーが利用可能で実行可能である必要があります」のように「すべき」という単語に注目してください。
  * > 予期されるメッセージ スタイルと推奨の根拠の詳細については、std::error モジュールのドキュメントの「Common Message Styles」のセクションを参照してください。
* [Common Message Styles](https://doc.rust-lang.org/std/error/index.html#common-message-styles)
  * > 人々がメッセージを期待する言い方には 2 つの一般的なスタイルがあります。
  * "expect as error message" (「エラー メッセージとして期待」)
    * メッセージを使用してパニックに遭遇したユーザーに情報を提示
  * "expect as precondition" (「前提条件として期待」)
     メッセージを使用してパニックをデバッグしている開発者に情報を提示

例にあるのは`std::env::var("IMPORTANT_PATH")`に対するこういうメッセージ。

* `IMPORTANT_PATH`変数が設定されていない
* `IMPORTANT_PATH`変数は`wrapper_script.sh`で設定されるべき

前者は「ソースコード見ればわかるよ」ということなんだと思う。
`.expect()`はその場でpanicするからソースコードの位置もわかるのだからそんな情報を出しても無駄だから、
追加できるとしたらここで期待する動作について書くことなのだろう。

## 私のログルール(暫定)

これはpanicしてソースコードの位置がわかることが前提となっているはずだ。  
一般的には、エラーが起きて自分で判断できないのであれば上の方まで伝搬させ、
panicさせずなるべくアプリの上の方でエラーをユーザーに見せて終わらせることが多いのではなかろうか。
`main() -> Result<>`でエラーを返してメッセージはRust側に任せるかもしれないが。

panicしないログはユーザーに見せないものなのでそういう制約は気にせず、
開発者がストレスなく解析するために役立つ情報が載っているのがよいはずだ。
まあ、ストレスなく解析するというのは無理かもしれんが。。。

複数の人で作るときは、先にログをどうするか決めておかないと
実装した人によってログのフォーマットやレベルが違うとイライラするだろう。

「何を当たり前な」と思うかもしれないが、エラー発生＝CPUリセットという世界だとそんなにログについて意識合わせしなかったりするのだ。
システムを作るとき、私はそういうのに極力かかわらずに慣れた人にお任せしたいね。
私にさせてはダメだ。  
今回は、私が一人で実装するときの指針がないので決めているだけだ。

### エラーの箇所をログだけでわかるようにしたい

ソースコードを見ればエラーの種類がわかると言っても、ログファイルを見てファイル名と行番号だけしかなかったら腹が立つだろう。
なので、タグジャンプできるようにファイル名と行番号は出力しつつ、エラーの内容も文字列だけで多少はわかるようにしておきたい。
そして、気になったログだけタグジャンプして見に行くのが理想だ。

場所がわかっても、この値がこうだからエラーだというような情報があれば出力したい。
そうなると`&'static str`よりも`String`になるのか。  
最初の頃はファイル名や行番号の出力ができると思っていなかったのでユニークな文字列を埋め込んでいたので`&'static str`を使っていた。
たとえば`reason`がそうだ。

```rust
    #[error("WinCode read error({source}): {reason}")]
    WinCodeRead {
        reason: &'static str,
        #[source]
        source: wincode::ReadError,
    },
```

C言語では、`__FILE__`と`__LINE__`だけしかなかったのだが、GNU拡張で`__FUNCTION__`があったし C99からだったか`__func__`が追加されて、
現在の関数名もログに出せるようになった。  
Rustで外部クレート無しで関数名を出すのは面倒なようだ。
tracingでも出せるようだが、`#[instrument]`を付けたところだけになるとかなんとか。調査不足だ。

まあ、それならそれでよいか。

### sourceを#[error]に含めるのは場合によりけり

以前`source`を`#[error]`に入れると長いので止めるという話をした。

* [rust: thiserrorはsourceをDisplayすると長すぎる - hiro99ma blog](https://blog.hirokuma.work/2026/07/20260703-rust.html)

このログは良かったのだが、他クレートのエラーはログに出したいと思った。
自分で実装しているエラーはわかるのでいらない。。。

が、私が作ったクレートを知らない人が使うという可能性もなくはない。
その人にとっては私の実装を知りたくもなかろう。

`source`の1つ下の階層だけ見せてくれれば良いのだが、そういうのはあまり簡単ではなさそう。
ログ出力するんだからもういいんじゃないのか、だんだんと弱気になってくる。。。

ログのメッセージと`#[error]`をまぜこぜで考えているのが良くないのか。  
`#[error]`の内容はわかりやすさよりも簡潔さが優先で良い気がする。
`#[error("Configuration error occurred")]`よりも`#[error("Configuration error")]`くらいでよい。
`#[error("Configuration")]`でもよいか？ 昔なら`#[error("config err)]`か`#[error("conf err)]`だったな。

[thiserrorのページ](https://docs.rs/thiserror/latest/thiserror/)にはこういう例もあった。
エラーの内容を説明しているし、値も見せているが、エラーが起きた、みたいな文言はない。
ここは比較的文章になっているがそうではない例もあった。
全体的に小文字始まりだったのは`source`で他のメッセージにも使われる可能性があるからか？ golangなんかはそういう感じだったような。

```rust
#[derive(Error, Debug)]
pub enum Error {
    #[error("first letter must be lowercase but was {:?}", first_char(.0))]
    WrongCase(String),
    #[error("invalid index {idx}, expected at least {} and at most {}", .limits.lo, .limits.hi)]
    OutOfBounds { idx: usize, limits: Limits },
}
```

英語が苦手なこともあり、文章を作りたくないのだ。それくらいは許されそうだ。

### tracingの出力例

`tracing_subscrier`で設定すれば`file!()`や`line!()`を付けなくても出力された。
わざわざマクロを作ってまで埋め込んだのだが・・・意味がなかったか・・・。

`#[instrument]`を付けておくと関数名も出力された。
`main()`の前につけておけばよいかと思ったがそれではダメだった。

```rust
use std::{error::Error, result::Result};

use tracing::{info, instrument};

#[derive(thiserror::Error, Debug)]
pub enum DataStoreError {
    #[error("invalid header (expected {expected:?}, found {found:?})")]
    InvalidHeader { expected: String, found: String },
}

macro_rules! mylog {
    ($err_variant:expr) => {
        info!("{}", $err_variant)
    };
}

#[instrument]
fn open() -> Result<(), DataStoreError> {
    mylog!("文字列だ");
    Err(DataStoreError::InvalidHeader {
        expected: "こうであってほしい".to_string(),
        found: "こうだった".to_string(),
    })
}

fn main() {
    tracing_subscriber::fmt::Subscriber::builder()
        .with_file(true) // ファイル名を出力するかどうか(フルパス)
        .with_line_number(true) // 行番号を出力するかどうか
        .with_max_level(tracing::Level::TRACE) // ログ出力する最大レベル(これより下は出力されない)
        .init();

    open().unwrap_or_else(|e| {
        eprintln!("{}", e);
        if let Some(err) = e.source() {
            eprintln!("base error: {}", err);
        }
    });
}
```

```log
2026-07-05T04:44:21.772304Z  INFO open: hello: src/main.rs:19: 文字列だ
invalid header (expected "こうであってほしい", found "こうだった")
```

`#[instrument]`はなんとなくログの負荷がかかりそうだが、呼び出し頻度が高くなければそこまではないらしい。  
しかしクレートだとあまり`tracing`使ってますという色を出したくないのではなかろうか。

Geminiに聞くと、あきらめて`tracing`を使えって感じの返事だった。
非同期が絡んでくると`log`では力不足というかなんというか。

どうなんかなあ、それでいいんかなぁ。。。  
アプリ側でどれを使うか決めたらクレートもそれに従うだけというのがありがたいのだがなあ。

## 方針

* `tracing`を使う
  * ファイル名、行番号、関数名がログに出せるのならもう仕方ないのではないか
  * 関数名が出力されたらログメッセージだけ見ても推測しやすくなることを期待したい
* 文章にしなくて良い
  * 文章のほうがよいならその限りではない
* ログを出したい箇所で`format!()`などで文字列を作るよりは、構造体にしてフィールドで値を与えて`#[error]`で文字列にする
  * `String`で渡すほうが変更しやすくて楽ではあるのだが、表現がバラける心配がある
  * 全部そうできるとは思えないので臨機応変に。。。
* `#[error]`のメッセージをそのまま`error!()`のログに出してもいいんじゃないのか
  * `Err`とログと、同じような内容のものを2つ作るのが苦痛なのだよ

### 例1

自作のライブラリを変更した。

最初のログは、発生原因とも言えるのでまあよかろう。  
2行目は単に経由しているだけであるが、それでもそこから呼び出したことによってエラーが起きているのだから経路情報としてはいるだろう。
条件分岐によって通ったりしたのなら、そういう情報を出力するのも良い。  
ただここでは特にそういうものがなく、この関数からこちらの関数を呼び出したということを`"load: ElectrumRpc::new"`を自分で埋め込んでいるだけだ。

```log
2026-07-05T04:38:54.560974Z ERROR btc_wallet::electrum: src/electrum.rs:28: Electrum client error(Connection refused (os error 111)): config.server=tcp://127.0.0.1:50001)

関数名やメソッド名を出力するには`#[instrument]`を付けるのだったなと、経路のメソッド前に`#[instrument(skip_all)]`を付けた。
`skip_all`なのはたぶん引数か戻り値かその辺に`Debug`を持っていなくて出力できない要素があったからだ。

この処理は`lib.rs`の`load()`から`electrum.rs`の`new()`を呼び出してエラーが起きていて、ログにもエラーレベル出力の次にその情報がある。

2026-07-05T11:10:17.907359Z ERROR load: btc_wallet: /home/hirokuma/Bitcoin/rust/rust-btc-keypath-wallet/src/lib.rs:224: backend error: load: ElectrumRpc::new
2026-07-05T11:10:17.907498Z ERROR sample: examples/sample.rs:46: wallet: backend error: load: ElectrumRpc::new
```

これって関数名が出力されているというよりは、span名をつけないと関数名などが自動で採用されているだけじゃなかろうか？  
`new()`の方に`#[instrument]`を書かないと出力されるのは "load" だけになり、`new()`の方も"load"になるのだ。
`tracing`のspanは入口から出口までなどのコンテキストを作るものなので、もしspan名が関数名と同じになるならそういう動作になるはずだ。

#### 何が失敗したのか、何で失敗したのか

* 「ファイルの読み込みに失敗しました」
* 「ファイルがありません」
* 「configファイルの読み込みに失敗しました」
* 「configファイルがありません」

最初の2つはエラーの情報から出力できそうだが、残りの2つは他の情報も追加しないと表現できない。
ファイル名があればよいかもしれないが、`test.dat`の読み込みに失敗したと言われてもそれが何なのかわからないかもしれない。

### エラーログ情報の種類

トレースログなどはもういい。エラーログだけに限定して考えよう。
限定しなくても良さそうな気がするが、あれこれ考えたくないのだ。

大きくは、物理的な情報とそれ以外に分けられる。  
物理的というか、コンパイルした時点で直接結びついている静的な情報だ。
具体的にはこれらだ。

* ファイル名(ファイルパス)
* 行番号
* モジュール名？
* 関数名？

最近の言語だと無名関数などがあるので関数名は含まれないかも。
モジュール名はRustであればコンパイル時に確定していそうな気がする。  
相対パスだとファイルパスだけだと弱いかもしれないが、まあそういう細かいことは良いだろう。
`tracing`のログでも最初の3つは出力できるので、物理的というか静的というかそういう情報なはずだ。
これらはいつでも出力できるので、あまり考えなくてよい。

次はエラーそのものの情報。  
自分で判断してエラーにした場合や、自分で実装していない外部の関数を呼び出してその結果がエラーだった場合は、
そのエラーは一番震源に近いエラー情報だ。  
ここでは「どういうエラーになったのか」と「なぜエラーになったのか」を見せたいだろう。
前者はまだ出力しやすいが後者は難しいかもしれない。
自分でエラーにしたのであれば説明できるが、関数を呼び出してエラーになったら自分では説明できないだろう。  
できそうなのは、その関数を呼出したパラメータを出力することだ。
これはその箇所でしか得られない情報なので見たいことが多いのだが、所有権を渡すパラメータだとエラーとして出力できないかも？  
エラーにした関数がエラーログに出してくれればよいが、期待しすぎてはダメだろう。
ネットワークエラーでサーバーに接続できないときでも「Connection refused (os error 111)」くらいしか返さないこともあるから、
むしろ過剰には返さないようにしているのかもしれない。  
"Connection refused"は英語の文法みたいに考えると、「接続は、失敗した」でS+Vで、"(os error 111)"は単なる補足情報なのか。

同じように考えると、自分が引数としてもらったものをそのまま使うだけならログとして見せる必要がないのかもしれない。
とはいえ`config`のように構造体を渡している場合はどれを使っているのか上位側もわからないかもしれないので見せたほうが親切か。

この場合はBitcoinバックエンド関係の関数だったので、接続に失敗すればBitcoinバックエンドへの接続に失敗したのだとわかる。
しかしファイルオープンだった場合、それがテキストエディタで編集するファイルのオープンの場合と設定ファイルをオープンする場合では違うメッセージにしたい。
それができるのはエラーになった箇所よりもアプリ側のどこかになる。

あまりむずかしく考えず、自分の関数では自分の分かる範囲で具体的な情報を付けて上に投げる、というだけでよい気がしてきた。

* 静的な情報
  * ファイル名(ファイルパス)
  * 行番号
  * モジュール名？
* 具体的な情報
  * エラーの理由
    * エラーの発火点なら何をチェックしてエラーにしたのかを構造体のメンバーにすれば良い
    * 関数呼び出しでエラーになったなら、そのエラー情報と渡したパラメータ
  * 何をしようとしたのか(optional)
    * 具体的に書ける場合だけでよい
    * 「ファイル読み込み」ではなく「設定ファイル読み込み」のような

### 構造体をログに与える

`error!(msg)`として扱っていたが、`error!(a=aaa, b=bbb, ..., msg)`のようにラベルとデータのセットを渡すことができる。
ここで`err`の値を渡すことで`#[error]`に文字列として組み込んでいたデータをなくすこともできる。

これは悩ましい。。。  
構造体の中に組み込まないと`#[error]`で使えないのでそうしていたが、
メッセージを短くして構造体を見るようにするのも魅力的だ。

そうなると、エラーが最初に発生したメッセージが重要だ。
その段階でユーザが見てもよいようなデータにして、
エラーを戻すところでは付加情報があれば構造体に突っ込むだけにするのがよいのか。

「何をしようとしたのか」を具体的な情報としてほしいのだが、それはログとしてほしいだけであって、
実装の中でその値を使ってなにかするわけではない。  
その情報を値として組み込まなくて済むのはよいことだと思う。

その代わり、ログとして出力する手段を考え直す必要がある。
今考えているのはこういうマクロ。

```rust
// log with parameter
#[macro_export]
macro_rules! log_err_wp {
    ($err_variant:expr, $msg:expr, $($key:ident = $val:expr),* $(,)?) => {{
        let err = $err_variant;
        error!(error = ?err, $($key = %$val),*, $msg);
        err
    }};
}

#[macro_export]
macro_rules! log_err {
    ($err_variant:expr, $msg:expr) => {{
        let err = $err_variant;
        error!(error = ?err, $msg);
        err
    }};
}
```
