あまりログの出し方というものに気を配って生きてこなかった。  
見たいものをそのままログにしていけば良いと思っていたのだが、
自分でログをいちいち埋め込まなくても出力できるといわれると悩んでしまう。
自動的にできるものはそうして、機械的にできるものはそうして、とにかく楽したいのだ。

エラーを返すことだけが目的のアプリエラーは`anyhow`でいいと思う。
`anyhow`だと情報が減るのだが、`anyhow`を使わないといけないということはその関数の中でいろいろなエラーが返ってきているということで、
それを一緒くたにしてよいのであれば情報が減ってもよいと思うのだ。

crateの場合はアプリ側にどうしたいのか判断してほしいので、何も考えずに情報を残して返してしまいたい。

## 暫定方針

`tracing`を使う前提で、なるべく`thiserror`のエラーメッセージだけにするならこんな感じだろうか。

* メッセージ本体は、小文字始まり、終わりに句読点を使わない、というスタイル
  * golangと同じようなルールでよさそう。理由も同じ。
  * golangのようにwarningは出ない
* 静的な情報(`tracing_subscriber`の設定で済む)
  * ファイル名(ファイルパス)
  * 行番号
  * モジュール名？
* 具体的な情報
  * エラーの理由(`thiserror`など)
    * エラーの発火点なら何をチェックしてエラーにしたのかを構造体のメンバーにすれば良い
    * 関数呼び出しでエラーになったなら、そのエラー情報と渡したパラメータ
    * `{source}`はなるべく`#[error]`に入れない
      * `error!(error=?e, "message...")`などとして`err`を構造体として見てもらう
    * `{source}`を`#[error]`に書くときは一番最後にコロンで区切って置くのが良いと思う
      * `#[error("message...: {source}")]`のような感じで
      * `source`は過去の分が連なっていくので途中に置くと見づらかった
    * なんでもかんでも`#[error]`に書くべきではない
      * 引数にしていた1つのファイル名やURLのような関係がかなり深いパラメータは載せるとよさそう
  * 何をしようとしたのか(optional)
    * 具体的に書ける場合だけでよい
    * 「ファイル読み込み」ではなく「設定ファイル読み込み」のような

`tracing`のspanはまだ決めきれない。
クレートの頭に付けておけばよいような気もするし、アプリ側の呼び出し頻度で負荷が変わるだろうからアプリ側だけでやっておけばよい気もするし。
今のところ後者に傾いている。

`#[from]`は追加情報を載せられないので、optionalな情報を載せたい場合は使用できない。
あと、自動変換になるので便利なのだが、そのエラーを返す箇所がなくなったとしてもコンパイラがわからない残り続けるという弱点がある。  

`#[from]`を使わないという場合は`.map_err`で変換することになる。
`.map_err`の中には文が書けるので、そこでエラーログを出すことができる。`#[from]`で自動変換されるとそれはできないと思う。  
`#[from]`でせっかく実装量を減らせたのに・・・という気持ちになるだろうから、場合によりけりかもしれない。

`#[error(transparent)]`は`#[from]`でしか使用できないが、なるべく使わずにいようと思う。
薄いラッパーだとそのまま透過させるのはありではないかな。
`cargo clippy`が文句を言わなければよいか、くらいの判断だ。

### 構造体

`error!(label=abc, label=def, "message...")`のように書くことで、文字列にしなくても値がログとともに出力できる。  
こういうマクロを作っておくと、`Err`のデータには含めないがログには出力したいという場合でも対応できる。
その代わり、エラーメッセージをそれぞれ書かないといけないので手間は増える。

```rust
// log with parameter
#[macro_export]
macro_rules! log_err {
    ($err_variant:expr, $msg:expr, $($key:ident = $val:expr),* $(,)?) => {{
        let err = $err_variant;
        error!(error = ?err, $($key = %$val),*, $msg);
        err
    }};

    ($err_variant:expr, $msg:expr) => {{
        let err = $err_variant;
        error!(error = ?err, $msg);
        err
    }};
}

...
  return my_func.map_err(|e| log_err!(MyError{ param: "name", source: e}, "my_func", name = name));
```

拡張トレイトでログを出力させると実際の行番号ではなくトレイトが実装された箇所になったりする。
それは`#[track_caller]`で対応できるらしいので、そちらを使うのもありかもしれん。

### あきらめて.map_errする

`#[from]`すると`.map_err`しなくなり、ログを出力するタイミングがなくなる。
なくなるわけではないのだが、「`?`で済むのにエラーログのためだけに実装したくない」という気分になってしまう。  
`#[source]`だと`.map_err`しないとコンパイルエラーになるのであきらめがついた。

拡張トレイトを作って`.log_err_with()`を後ろに引っ付ける例をGeminiが作ってくれたのだが試していない。
こちらだと`#[from]`でいけそうだし、`.map_err`に比べると楽そうだ。

```rust
pub trait LogErrorExt<T, E> {
    fn log_err_with<F, V>(self, msg: &str, f: F) -> Self
    where
        F: FnOnce() -> V,
        V: std::fmt::Debug;
}

impl<T, E: std::fmt::Debug> LogErrorExt<T, E> for Result<T, E> {
    // 💡 ここに付与することで、呼び出し元の行番号が tracing に引き継がれる
    #[track_caller]
    fn log_err_with<F, V>(self, msg: &str, f: F) -> Self
    where
        F: FnOnce() -> V,
        V: std::fmt::Debug,
    {
        if let Err(ref e) = self {
            // #[track_caller] のおかげで、ここの行ではなく、
            // .log_err_with() を呼び出したコードの行番号が出力される
            tracing::error!(error = ?e, context = ?f(), "{msg}");
        }
        self
    }
}
```

### 迷いあり

関数呼び出し時の引数をログに出すかどうかにちょっと迷いあり。  
呼び出した先でも出力しそうな気がするので、ログがうざったくならないかを気にしているのだ。

でも、相手が出力しないなら自分しか出力するところがないのでとりあえず出すのか。
構造体全部、みたいな出力の仕方はさすがに避けたいところだ。
