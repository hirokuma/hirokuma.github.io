数カ月Rustでコードを書いているが、書けはするもののRustの良さをあまり活かせていないことに気づくようになってきた。
無知の知というかなんというか。

コンパイラがエラーや警告してくれたり `cargo clippy` で教えてくれる内容には対応できるのだが、
そうでないものはけっこうC言語っぽい書き方をしているような気がする。

せっかく最近の言語を使っているのだから、生産性がどうのこうのというのはさておきコーディングしている人が楽になってしかるべきだと思うのだ。
楽をしたいのだよ、楽を。

## iterでfilterして新しいVecを作りつつ、あるフィールドの合計を求めたい

例えばこういう `Vec<Abc>` があり、`value` がある値以上の新しい `Vec<Abc>` を作る。
ついでにそのときの `value` の合計も求めたい。

```rust
struct Abc {
  value: u32,
  name: String,
}
```

なんかそういう技もあるかもしれんとGeminiに聞いてみると。。。

* 2回 `.iter()` で回す
* `for` で回す

と、まあそうだよねという回答が返ってきた。  
それだけかと思ったが、「データ数が数万程度だったら2回回していいんじゃないの」という補足がついていた。
パフォーマンスが極限まで求められるなら `for` とも。

えー、数万ってそういうレベルのデータ数なの？？  
2回回すなんてもってのほか、くらいな回答だと思っていたのに驚きだ。

ChatGPTは `for` 一択だった。
`for` で回して `push()` していくことになるので `Vec::with_capacity()` でメモリの再確保が起きないようにできるならなおよし、というところだった。

さて、最近使い始めたばかりの LM Studio にも聞いてみよう。qwen/qwen2.5-coder-14b という 8.37GB もある LLM だ。  
この子は2回回す方を進めてきた。
ループを回し過ぎでは？と質問すると `fold()` を提案してきた。GeminiもChatGPTも `fold()` は名前を上げつつすすめるというほどではなかったのだ。

どれも間違いではない。
ループ数やフィルタされる数が少ないか多いかでも変わってくるだろう。  
個人の好みで言えば、`fold()` を使うくらいなら `for` の方を使うだろうな。新しく覚えなくて良いし。

`iter()` で2回回すのはループ数が多くなって嫌だなあと思うことを除けばわかりやすいかもしれない。
最適化によってこのくらいだったら意図を読んでくれるかもしれん。
「楽をする」という観点からすると2回やるのがいいのかなあ。どうなのかなあ。。。
