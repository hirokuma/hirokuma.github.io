### HTLCスクリプトのアドレス

* [code](https://github.com/hirokuma/bdk-script-example2/blob/37f43a8f51b61cc70ec1dd7cffc46c133542f48a/src/main.rs#L46-L86)

前半の山場だ。

seedを作って、それを元にして拡張秘密鍵作って `bdk_wallet::Wallet` を作ってフルスキャン。  
わざわざseedを文字列にして渡しているのは最初の実装の名残で深い意味はない。  
`Wallet` を永続化した場合、それを読み込んだだけでは拡張公開鍵があるだけで署名はできない。
別途拡張秘密鍵を保持しないといけないのだ。今回の本題ではないので省略。

今回はHTLCでどちらのルートを通ってもこの作ったウォレットに送金させるので、そのアドレスを作る。  
実際にスクリプトやトランザクションで使うのはscriptPubKeyなのであとで変換する。
先に必要なデータを作るのでそういう順序になったと思ってくれ。

アドレスの次に作っているのがHTLCの鍵のような扱いになるpreimage。
ついでにpayment_hashも作る。特に言うことはない。

次は、先程作ったアドレス2つをそれぞれ `descriptor!{}` の `pk()` に与えられる形式に変換。
[convert_descriptor()](https://github.com/hirokuma/bdk-script-example2/blob/37f43a8f51b61cc70ec1dd7cffc46c133542f48a/src/main.rs#L386-L402)という関数を作った。
他のところでもそうだが、`AddressInfo` を作っているがほとんど使っていない。
使っているのはウォレットとアドレスがExternalかInternalか、そして何番目かという情報だ。
Electrum Serverへの問い合わせでscriptPubKeyを使うというのもある。そのくらいなのである。  
アドレス文字列は人間用ですな。

`convert_descriptor()` は `Wallet` のExternal側についてindex番目の派生をさせて、internal pubkey(X座標だけ)を取り出している。
こういう、internal pubkeyがいるのだったかtweaked pubkeyがいるんだったかでしばしば迷う。

そしてHTLCスクリプトはこうなる。
`descriptor!{}` が返す3つのtupleのうち、今回使うのは1番目の `miniscript::Descriptor<DescriptorPublicKey>` である。

```rust
let (descriptor, key_map, networks) = descriptor!(
    tr(
        intr_ext,
        {
            and_v(v:sha256(payment_hash), pk(addr1_xonly)),
            and_v(v:older(16),            pk(addr2_xonly))
        }
    )
)?;
```

スクリプトを[miniscript](https://bitcoin.sipa.be/miniscript/)の"Analyze a Miniscript"で確認しておく。  
上側のスクリプトを整理して `and_v(v:sha256(H), pk(A))`。
これをanalyzeすると`OP_SIZE <20> OP_EQUALVERIFY OP_SHA256 <H> OP_EQUALVERIFY <A> OP_CHECKSIG` と返ってくる。
`<20>` は16進数で、まずはスタックデータのサイズをチェックして、
それをSHA256して一致したら署名チェックするようになっている。

これをアドレスにしているのだが、`htlc_descriptor` を `.at_derivation_index()` するのはなぜだろう？
`htlc_descriptor` は[`enum miniscript::descriptor::Descriptor<DescriptorPublicKey>`](https://docs.rs/miniscript/12.3.5/miniscript/descriptor/enum.Descriptor.html)である。
今回は `Tr(Tr<Pk>)` になっている。  
[at_derivation_index()](https://docs.rs/miniscript/12.3.5/miniscript/descriptor/enum.Descriptor.html#method.at_derivation_index)はワイルドカードの置き換えだそうだ。
ならば今回はなくてもよさそうだが、戻り値が `Descriptor<DefiniteDescriptorKey>` になる。
そちらでないと `.address()` が使えないのだ。

考え方としてはこんなものか。

* `descriptor!{}` の戻り値の段階ではそのDescriptorはまだワイルドカードなどが入っている可能性がある
* そこから確定(definite)したDescriptorに変換するために `.at_derivation_index()` を使う
* アドレスはdefiniteしたDescriptorから取得できる

今回はワイルドカードがないので `.at_derivation_index(0)` だろうと `.at_derivation_index(100)` だろうとなんでもよいはずだ。
(`Wallet` から派生したアドレスだと `AddressInfo::index` を使ったりする。[convert_descirptor()](https://github.com/hirokuma/bdk-script-example2/blob/37f43a8f51b61cc70ec1dd7cffc46c133542f48a/src/main.rs#L386)参照。)

### definite_htlc

戻り値が `Descriptor<PublicKey>` になる関数はいくつかあるが、いずれも `&Secp256k1` を必要としている。
[convert_descriptor()](https://github.com/hirokuma/bdk-script-example2/blob/37f43a8f51b61cc70ec1dd7cffc46c133542f48a/src/main.rs#L386)もそうなっている。
libsecp256k1を使っているのかもしれない。

`definite_htlc`は[その下](https://github.com/hirokuma/bdk-script-example2/blob/37f43a8f51b61cc70ec1dd7cffc46c133542f48a/src/main.rs#L137)で使われているが、
ここは `tr.spend_info()` があればよいだけなので `htlc_derived`でも使えた。
`htlc_descriptor` では `ToPublicKey()` トレイトがないのでダメらしい。
まだdefiniteな形ではないのでpublic keyを導くことができない、という考え方で良いのかな？

この`definite_htlc`が必要になるのは最後にwitness stackを得る`get_satisfaction(satisfier)`のところだけだった。

### Satisfier

* [code](https://github.com/hirokuma/bdk-script-example2/blob/37f43a8f51b61cc70ec1dd7cffc46c133542f48a/src/main.rs#L184-L251)

P2TRのルールに従うとHTLCのpreimageルートを解くために必要なwitnessはこうなる。

* addr1の鍵でのシュノア署名
* preimage
* preimageルートのスクリプト
* control block

これまでのデータがあればwitness stackを作ることができる。  
単にwitness stackを作ると間違いがあるかもしれないのでSatisfierを使って安全にやりましょう、というところか。

satisfierというのはシュノア署名やP2TR、[BIP-379](https://github.com/bitcoin/bips/blob/master/bip-0379.md)に出てこない。  
おそらくsipa実装の[struct Satisfier](https://github.com/sipa/miniscript/blob/6806dfb15a1fafabf7dd28aae3c9d2bc49db01f1/bitcoin/test/miniscript_tests.cpp#L163)から来たのだろう。

[miniscript::satisfy::Satisfier](https://docs.rs/miniscript/12.3.5/miniscript/miniscript/satisfy/trait.Satisfier.html)というトレイトである。  

> 署名、ハッシュ プリイメージなどのルックアップ テーブルを記述するトレイト。
> すべてのメソッドには、クエリごとに単に None を返すデフォルトの実装があります。
> ユーザーは、データを持っているメソッドをオーバーライドすることが期待されます。

だそうだ。
ルックアップテーブルというと小難しいが、それらの値を取得するメソッドを持つトレイトを用意したから中身を埋めろということだ。
あとはライブラリ側がそれらを使ってうまいことやるんでね、と解釈した。

`Satisfier`トレイトのページを見てどれを自分が実装しないといかんのかわかるのだろうか？  
今回はこの3つが実装されていた。
スクリプトが`OP_SHA256`なので`lookup_sha256()`を実装して、あとはP2TRなので`tap_`系、internal keyではないので`lookup_tap_leaf_script_sig()`というところか。

* `lookup_tap_leaf_script_sig()`
* `lookup_sha256()`
* `lookup_tap_control_block_map()`

なんとなくだが、これを自分で実装するのは難しい気がする。
中身は1行程度なのだが、これが思いつくかと言うとまったく自信はない。  
ありがちなスクリプトについてはSatisfierもセットにして作ってあると便利なのだがなあ。
rust-bitcoinとminiscriptが使われていてbdk_walletは出てこないように見える。

ともかくHTLCのpreimageルート用に`ClaimSatisfier`を実装するまでが前半戦とすると、それを使ってwitness stackを作るのが後半戦だ。

`claim_pk`はpreimageルートのアドレスaddr1の公開鍵X座標`addr1_xonly`を`bitcoin::PublicKey`に変換したものだ。
コメントの「Taprootはeven parityと仮定」が不穏だ。

```rust
let pk = PublicKey::from(addr1_xonly.public_key(bdk_wallet::bitcoin::secp256k1::Parity::Even));
```

Taprooではxonly、つまり圧縮公開鍵だと"`02` or `03` + 32バイト(公開鍵X座標)"の先頭を削った32バイトだけを使うのだが、それだとBitcoin形式の公開鍵にならない。
`public_key()`は不足している先頭部分を`EVEN`か`ODD`でもらって付け足している。

* [source](https://docs.rs/secp256k1/0.31.1/src/secp256k1/key.rs.html#546-557)

secp256k1は`y^2 = x^3 + 7`という式なので、同じX座標に対してY座標は±な値になる。
実数だとそうなのだけどBitcoinでは有限体ということで？だから？とにかくmoduloな世界ということで`p-y`を使う。
`p`が奇数なので`p-y`は`y`の偶数・奇数が反転した結果になる。  

1. `x`から`y`を求める(`0 <= y < p`)
2. プレフィクスと`y`の組み合わせで最終的な値が決まる
   * `02`で`y`が偶数 → そのまま使う
   * `02`で`y`が奇数 → `p-y`
   * `03`で`y`が偶数 → `p-y`
   * `03`で`y`が奇数 → そのまま使う

などなど調べていたのだが、[BIP-340 Design](https://github.com/bitcoin/bips/blob/master/bip-0340.mediawiki#design)の"Implicit Y coordinates"にこう書いてある。

> 2.Implicitly choosing the Y coordinate that is even[6].

暗黙で偶数扱いなのだ。  
この記述、昔の私は見逃していたかもしれない。。。  
ともかくここについては「仮定」というよりも「仕様」だろう。

しかし`Wallet`から派生させているならそんなことしなくてもアドレスを作る仮定で公開鍵をもらってくればいいんじゃないかと思った。
調べてみたが`for_each_key()`でぐるぐる回して取ってくるので返って面倒そうだ。

```rust
let mut pk: Option<PublicKey> = None;
htlc_derived.for_each_key(|key| {
    pk = Some(key.to_public_key());
    false
});
let pk = pk.ok_or_else(|| anyhow::anyhow!("not found"))?;
```

あとは値を詰めていって`ClaimSatisfier`を作り、`get_satisfaction()`で`witness_stack`を取得し、最初の方に作った`spend_tx`のwitnessに詰める。
