Rustで自分が知らないクレートが出てきたときはほぼAIにやってもらったときだ。
`once_cell::Lazy::new()` もそうだった。

Rustは `static` 変数が静的に決まった値で初期化できるようになってないと使えないので回避するのに使ったりする。
他にもあると思うが、今回はそういう使い方だった。
単に `format!()` を使いたかっただけなんだけどね。。。
使わなくてもなんとかできたと思うが、もう面倒だったからいいや、と。

それから数ヶ月経ってAIが `static` な変数を作るときに `Lazy` を使っていて、これなんだっけと調べ直したのが今回の話だ。

## once_cell::sync::Lazy

このクレートのリポジトリはこちら。今日見ると[v1.21.4](https://github.com/matklad/once_cell/releases/tag/v1.21.4)が最新だった。

* [matklad/once_cell: Rust library for single assignment cells and lazy statics without macros](https://github.com/matklad/once_cell)

## std::sync::LazyLock

Geminiに `Lazy` について質問すると、概要とともに `std::sync::OnceLock` や `std::sync::LazyLock` もありますよと紹介してきた。

* `std::sync::OnceLock`: Rust v1.70.0以降
  * [OnceLock in std::sync - Rust](https://doc.rust-lang.org/std/sync/struct.OnceLock.html)
* `std::sync::LazyLock`: Rust v1.80.0以降
  * [LazyLock in std::sync - Rust](https://doc.rust-lang.org/std/sync/struct.LazyLock.html)

`OnceLock` は `get_or_init()` を呼ぶが `Lazy` は定義時に書ける所が良いらしい。  
そして `LazyLock` は書き換えについては `Lazy::new()` を `LazyLock::new()` にするだけでよいみたい。
`OnceLock` は `LazyLock` のシンプル版

じゃあこれからは `std` の方だけでいいやんという気もするが、今までがんばってた matklad さんに対してこれで良いのかという気もしてしまう。
Rust v1.80.0のリリースが2024年7月25日なので1年半くらい経っている。やさぐれてないだろうか。。。

と心配したが matklad/once_cell の最新のv1.21.4は2026年3月12日と最近のことだ。
元気そうで安心したが、まだリリースすることがあるのだろうか？ そもそも違いはなんだろうか？

### LazyLockとLazyの違い

まず、Rustのバージョンに依存しない。
そして `Lazy` は `std` ではないので `no_std` 環境でも使うことができる。

`force()` がメソッドか関数かという違いもある。

`once_cell` の方にはスレッドセーフではない `Lazy` もある、とのことだったが `std::cell::LazyCell` が同じような立ち位置じゃないのかな？

ちょっと自分でそういう状況にならないとわからんのだが、`once_cell` の方が歴史が長いしメソッドも色々あるので小回りがきくのかもしれない。
