作っているプログラムで、自作libが使っているクレートのエラーを面倒なので `anyhow::Result<T>` で処理していた。
が、やってるうちによろしくない雰囲気を感じたし、本などを読むと「`anyhow` はアプリくらいにしておこう」ということだった。
私もそう思うよ。

というわけで現在置換え作業をしている。  
まあ、もともとエラーの種別を気にしないで済んでいたから何でも良いのだけど、
せっかくなので `thiserror` でなるべく下側のエラーはそのまま上に返そうとしている。
`#[from]` が使えるなら `.map_err()` も書かないで済むから楽そうだ。

そう思ったのだが、意外とエラー型がよくわからないということに気づいた。

## ジェネリックを含む

今つまづいているのが、これ。

* [CreateParams in bdk_wallet - Rust](https://docs.rs/bdk_wallet/latest/bdk_wallet/struct.CreateParams.html#method.create_wallet)

`Result<T, E>` なのでエラー型は `CreateWithPersistError<P::Error>` である。
`P` はトレイトではなくジェネリックか？  
どうしたらよいんだ。。。

[CreateWithPersistError](https://docs.rs/bdk_wallet/latest/bdk_wallet/enum.CreateWithPersistError.html)でいう `<E>` のところだが、
`Persist(E)` 以外は使われなくても指定しないとダメなのだろう。  
永続化にSQLiteライブラリのrusqliteを使っているので、Deep Wikiにその旨を伝えてやってもらうと解決してくれた。  
`bdk_wallet::CreateWithPersistError<bdk_wallet::rusqlite::Error>` になる。長いわぁ。

* [deepwiki](https://deepwiki.com/search/createwalletthiserrorfromrusql_323e5e6a-82a6-40c7-bf21-f53719cf0bbc?mode=fast)

## map_err()はクロージャでなくても良い

`thiserror` で `#[from]` があっても `?` がないと自動では変換してくれない。
このあたりはまだ理解できていないところがありそうな気がする。

例えば、これはコンパイルエラーにならない。

```rust
use std::result::Result;
use thiserror::Error;

#[derive(Error, Debug)]
pub enum MyError {
    #[error("An error occurred: {0}")]
    ParseError(#[from] std::num::ParseIntError),
}

fn main() -> Result<(), MyError> {
    let v: u64 = "hello".parse()?;
    println!("Parsed value: {}", v);
    Ok(())
}
```

しかしこれはダメだ。
`parse()` のエラー型は `ParseIntError` であって `MyError` ではないのだ。

```rust
use std::result::Result;
use thiserror::Error;

#[derive(Error, Debug)]
pub enum MyError {
    #[error("An error occurred: {0}")]
    ParseError(#[from] std::num::ParseIntError),
}

fn parse_value(s: &str) -> Result<u64, MyError> {
    s.parse() // ★ここ
}

fn main() -> Result<(), MyError> {
    let v: u64 = parse_value("hello")?;
    println!("Parsed value: {}", v);
    Ok(())
}
```

こういうときは `.map_err()` でエラーの方だけ変換する。  
私は `.map_err()` の中は必ずクロージャじゃないとダメだと思っていたのだが、
`#[from]` があるからなのかこれだけで通る。

```rust
    s.parse().map_err(MyError::from)
```

直接エラー名を書いてもよい。

```rust
    s.parse().map_err(MyError::ParseError)
```

型に厳しいRustなのだが、こうやってよくわからない受け入れ方ができることがある。
引数の型違いを定義できるオーバーロードがあるんじゃないかと思ってしまうくらいだ。

ここでは `.map_err()` で受け付けるトレイト境界が `FnOnce(E) −> F` と関数を受け付けるからと、
`enum`の値(variant)は関数だからできるとのこと。  
確かにクロージャは関数だし、`enum`の値が関数扱いならそうなんだろう。

* [gemini](https://gemini.google.com/share/41757f281e59)

なんとなくクロージャは「クロージャ」というジャンルと思いこんでいたように思う。
だって縦棒2本使って囲んだところが引数扱いになるなんて言語は初めてだったのだよ。

## very large

今まで `anyhow::Result` だったところを `thiserror` に置き換えていったところ、`cargo clippy` でこういうwarningが出るようになった。

> warning: the `Err`-variant returned from this function is very large
> .....
> the largest variant contains at least 192 bytes

AIに質問ができるというのは便利なものだ。

* [gemini](https://gemini.google.com/share/d4d4143db83d)
  * `Result<T, E>`のサイズは `sizeof<T> + sizeof<E> + α` になる
  * このサイズはスタックに確保される
  * `thiserror` で `#[from]` があるとそれもスタックに入る
  * `enum` のサイズは全variantの合計ではなく最大のvariantに合わせられる

つまり1つでも大きいvariantがあるとそれ以外を返す箇所も引っ張られるということだ。
なお `String` は文字列自体はヒープに置かれるので文字長には影響されない。  
この辺は C/C++ と同じですな。

今の私の知識では、C/C++のポインタ渡しに相当するのが `&` 付きで、C++の参照っぽく渡すのが `Box<T>` と思っている。
まあ、 `&` を使うとライフタイムが出てきて面倒になりそうだったら `Box<T>` を使っているだけだが。  
Geminiがいう解決方法の `Box<T>` にするというのも、元データはコピーせずアドレスだけ使うということなんだろう。
でも元がスタックだったら同じなんじゃなかろうか？ そこはカウントされないのか。

それは後で考えるとして、この `Box<T>` で囲む方式だが、`#[from]` があって `?` を使っても不一致でエラーになる。
[ChatGPT](https://chatgpt.com/share/69f17315-bed0-83a7-9e93-6b0b29c058ec)や[Claude](https://claude.ai/share/fbc793bc-10c4-4278-b047-27bc61583cf6)に質問すると「それはできない」というのだが、Geminiは何度聞いても「できる」と言ってくる。
[DeepWiki](https://deepwiki.com/search/fromboxvariantbox_df6903e9-fc07-45fc-8c5b-1c2219860f0b?mode=fast)もできないという。  
私もそう思うのだが、Geminiにそれを理論付けて説明できるほどの知識はない。

何か変換するようなマクロがあるのではと探すと [thiserror-ext](https://docs.rs/thiserror-ext/latest/thiserror_ext/index.html) というものがあった。
これは個別のvariantを `Box` にするのではなくエラー型自体を `Box` などでまとめて新しい型を作るマクロのようだ。
`Box` 以外にも指定できるようだ。

便利そうだがAIたちの誰も言わなかった。Claudeさんも「あまり使われていないですね」とのこと。
まあ確かに、しょせんは戻り値なのでそんなに目くじらを立てて修正しないといけないというほどでもない気はする。
スタックメモリも組み込み開発ほど少ないということはないだろうし、よっぽど変なことをしなければ放置でも良さそうだ。

`thiserror-ext` を使ってみたが、自動では無理なところも出てくるので修正箇所もそこそこ発生した。  
それならサイズが大きいvariantのみ `Box` にしていったほうが楽だった。依存も増えないし。
どのvariantのサイズが大きいのかどうやって調べようかと考えていたが、clippyでwarningが出たのはサイズが大きいものだけだったようだ。
大量に出ていたので全部出ていたのかと思った。
