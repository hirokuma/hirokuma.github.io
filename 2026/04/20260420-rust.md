C言語は暗黙の型変換が行われやすいので、運用するコーディング規約で「明示的にキャストする」というルールを採用しているところがあるとかなんとか。
キャストすると今度はコンパイラによる警告がでなくなるし、最近のコンパイラは賢くなったのでそういう変なことをしなくてもよいのかもしれない。  
私が最初に仕事で使ったコンパイラは、まだ戻り値に `struct` がなかったり、もしかしたら `void` が採用される前の仕様だったような気がする。
最適化にバグがある可能性がけっこう高かったので禁止したり。  
懐かしい話だ。

さて、それからするとRustは本当に自動的な変換がほとんど行われない言語だ。
以前の、`Result<T,E>` が戻り値で `?` を使っていて `From` トレイト実装に一致するエラーがあれば自動的に使ってくれる、というのは珍しい話だ。
トレイトが関係すると、そういう自動変換に見えるようなルールが働きそうな気がする。

それ以外は `u32` の変数を `u64` に代入するのだって明示しないとエラーになるくらいには自動的にやってくれない。
だからかもしれないが、なんかやたらと変換関係の説明が多い。
`to_string` はまだわかりやすいが、`to_owned` だったり `into` だったり、`from` や `parse` もある。
そうかと思えば `as ～` でキャストできることもあるし、未だに全然把握できない。

* [型変換 - Rust By Example](https://doc.rust-lang.org/stable/rust-by-example/ja/conversion.html)

## 型推論

型の違いに厳しいおかげなのか、実装だけで型の推測をする型推論が得意だそうだ。

* [型推論 - Rust By Example](https://doc.rust-lang.org/stable/rust-by-example/ja/types/inference.html)

しかしどこでも推論してくれるわけではなく、たとえば関数の戻り値なんかは明示的に型がわかったとしても省略できない。
`struct` のフィールドもそうだ。

```rust
fn func() -> _ { // replace with the correct return type: `u32`
    128u32
}

struct Val {
    value: _, // the placeholder `_` is not allowed within types on item signatures for structs not allowed in type signatures
}

fn main() {
    let val = Val { value: func() };
    println!("value={}", val.value);
}
```

なので、ライブラリで戻り値が複雑な型が返ってくるけど知らずに使えるから便利～、と思っていても、
それを他のところでも使うので関数化しようとしたら「型は何だっけ？」になって困るのだ。
vscodeだとinlay hintとして出てくるのでダブルクリックすればよいのだが、
今度はその中に出てくる型が `use` しないと使えないので追加して。。。
