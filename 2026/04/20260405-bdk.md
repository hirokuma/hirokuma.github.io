[BDK](https://bitcoindevkit.org/)で、Electrum Serverをバックエンドにして [bdk_wallet](https://docs.rs/bdk_wallet/latest/bdk_wallet/) を使っている。  
一昔前は `bdk` というクレートを使った bdk v0系だったようだが、現在は `bdk_chain` や `bdk_electrum` などのクレートに分かれた v1系になっている。
AIに聞くとv0系での回答を返すことがあるので注意しよう。
どちらの系もrust-bitcoinがベースなのでエラーなく動いてしまったりするのだ。

ずっとアプリを動かしていたのだが、妙に時間がかかることがあって調べていた。
今のところ、bdk_wallet が内部で持っているトランザクション情報を保持する `TxGraph` が大きくなりすぎたのが原因じゃないかという予想だ。  
・・・と書くとコードを調べていたように思われそうだが、時間がかかっているメソッドを中心にAIへ質問していった結果というところである。

TxGraph 自体は [bdk_chain::tx_graph::TxGraph](https://docs.rs/bdk_chain/0.23.2/bdk_chain/tx_graph/struct.TxGraph.html)に定義がある。  
直接扱うことはまずなかろう。`PersistedWallet<Connection>` のインスタンス経由で使用されるので意識することはないと思う。
それだけに、どのメソッドを使うと時間がかかるのかわかりづらい。

以前 [SyncRequest](https://blog.hirokuma.work/2026/03/20260314-bdk.html) のことを書いたが、あれは同期に時間がかかるのをなんとかしたかったからだ。
TxGraph が大きくなったことが原因で `sync()` が遅くなったのだと思い込んでいたのだが、
よくよく時間を測ってみると `SyncRequest` を作るときに [Wallet::transactions()](https://docs.rs/bdk_wallet/latest/bdk_wallet/struct.Wallet.html#method.transactions) を呼び出して
unconfirm なトランザクションを探し出すということをやってて、この `transactions()` に TxGraph が関わっていて時間がかかっていたのだった。  
`sync()` は `SyncRequest` の内容によって時間がかかるからそちらにばかり気を取られていて、作ることに時間がかかるのは気に留めていなかった。
AIに作ってもらったのをよく調べずに使っていたのは良くなかったよ。。。  

`transactions()` に時間がかかるのだから `list_unspent()` も同様だし、
UTXO を必要とするトランザクション作成もそうだ。  
トランザクションを作成するときに INPUT を指定すればよいかというとそうでもなく、
[CoinSelectionAlgorithm](https://docs.rs/bdk_wallet/latest/bdk_wallet/coin_selection/trait.CoinSelectionAlgorithm.html) で `coin_select()` の引数に `optional_utxos` があるためそうもいかない。
[manually_selected_only()](https://docs.rs/bdk_wallet/latest/bdk_wallet/struct.TxBuilder.html#method.manually_selected_only) の場合はよいかもというAI回答があったがまだ試していない。

あと、何気なく使っていて時間がかかったのが、`balance()`。
よくある残高取得なので気軽に使っていたのだが、これも TxGraph を走査するからか時間がかかっていた。

## TxGraphは小さくできないらしい

ならばもうspentしたトランザクションを忘れるなどしてTxGraphを小さくすればいいんじゃないかと思うが、
TxGraphには全部を記録するという設計になっていて減らせないということだった。
`rusqlite` を re-export しているくらいだから直接 DB をいじることはできるかもしれんが、まあやりたくはないな。

そういうことをしたければ、パスフレーズなどによるウォレットの再構築をするのがよいだろうというAI回答。
それかもう自前で管理してしまえ、ということだった。  
同期は `full_scan()` や `sync()` の戻り値があるし、受信アドレスは TxGraph に関係ないのでそのまま使えばよいので、がんばればできそうだ。

そのがんばりをしたくないから bdk_wallet を使っているのだがね。。  
途中から bdk_wallet 無しでがんばる気も起きないので、メソッドを呼び出して時間がかかっている箇所は回避できそうなら別の方法を使うようにしている。
チェーン関係は `bdk_electrum` で Electrum Server に問い合わせるなどだ。
