## 基本スタイル

基本的な書き方はこの4パターン。

* [thiserror - Rust](https://docs.rs/thiserror/latest/thiserror/)

```toml
[dependencies]
thiserror = "2.0.18"
```

```rust
use thiserror::Error;

#[derive(Error, Debug)]
pub enum DataStoreError {
    // その1
    #[error("data store disconnected")]
    Disconnect(#[from] io::Error),

    // その2
    #[error("the data for key `{0}` is not available")]
    Redaction(String),

    // その3
    #[error("invalid header (expected {expected:?}, found {found:?})")]
    InvalidHeader {
        expected: String,
        found: String,
    },

    // その4
    #[error("unknown data store error")]
    Unknown,
}
```

### その1: fromを使う

`#[from]`を使うと元のエラー情報を含んだままのエラーにできる。  
また、型が一致したら自動的に変換してくれる。

```rust
use std::{error::Error, fs::File, io, result::Result};

use thiserror::Error;

#[derive(Error, Debug)]
pub enum DataStoreError {
    // その1
    #[error("data store disconnected")]
    Disconnect(#[from] io::Error),
}

fn open() -> Result<(), DataStoreError> {
    let _f =  File::open("abc")?;
    Ok(())
}

fn main() {
    open().unwrap_or_else(|e| {
        eprintln!("{}", e);
        if let Some(err) = e.source()  {
            eprintln!("base error: {}", err);
        }
        eprintln!("-----------------------------");
        eprintln!("{:#?}", e);
    });
}
```

実行すると、`"{}"`では`#[error()]`そのものが、`.source()`で`Some()`が返ってきたら元になったエラーが取得できる。

```log
data store disconnected
base error: No such file or directory (os error 2)
-----------------------------
Disconnect(
    Os {
        code: 2,
        kind: NotFound,
        message: "No such file or directory",
    },
)
```

エラーメッセージを`#[transparent]`で受け流すのはあまりよろしくないそうなので、面倒だが`#[error()]`でなにかメッセージを作ったほうが良いだろう。

`#[from]`は便利なのだがエラーメッセージ以外はいじることができなさそうだ。  

### その2: タプル構造体

これはエラーメッセージを補強するデータとしてタプル構造体を使っている、でいいのかな。
タプルの最初が`{0}`、次が`{1}`のように使える。

```rust
use std::{error::Error, result::Result};

#[derive(thiserror::Error, Debug)]
pub enum DataStoreError {
    // その2
    #[error("the data for key `{0}` is not available")]
    Redaction(String),
}

fn open() -> Result<(), DataStoreError> {
    Err(DataStoreError::Redaction(format!("まったくもう")))
}

fn main() {
    open().unwrap_or_else(|e| {
        eprintln!("{}", e);
        if let Some(err) = e.source()  {
            eprintln!("base error: {}", err);
        }
        eprintln!("-----------------------------");
        eprintln!("{:#?}", e);
    });
}
```

```log
the data for key `まったくもう` is not available
-----------------------------
Redaction(
    "まったくもう",
)
```

### その3: 構造体

その2に近いが、こちらは構造体。お好みで？

```rust
use std::{error::Error, result::Result};

#[derive(thiserror::Error, Debug)]
pub enum DataStoreError {
    // // その1
    // #[error("data store disconnected")]
    // Disconnect(#[from] io::Error),

    // // その2
    // #[error("the data for key `{0}` is not available")]
    // Redaction(String),

    // その3
    #[error("invalid header (expected {expected:?}, found {found:?})")]
    InvalidHeader {
        expected: String,
        found: String,
    },

    // // その4
    // #[error("unknown data store error")]
    // Unknown,
}

fn open() -> Result<(), DataStoreError> {
    Err(DataStoreError::InvalidHeader{expected: "こうであってほしい".to_string(), found: "こうだった".to_string()})
}

fn main() {
    open().unwrap_or_else(|e| {
        eprintln!("{}", e);
        if let Some(err) = e.source()  {
            eprintln!("base error: {}", err);
        }
        eprintln!("-----------------------------");
        eprintln!("{:#?}", e);
    });
}
```

```log
invalid header (expected "こうであってほしい", found "こうだった")
-----------------------------
InvalidHeader {
    expected: "こうであってほしい",
    found: "こうだった",
}
```

構造体の場合は`#[source]`をつけるなりフィールド名を`source`にするなりで`#[from]`のように
フィールドの方が`Error`で終わる名前(`Error`のみも)でもそう判定されるそうだ。  
`#[from]`は便利だけど情報がちょっと足りないというときに使うことになるだろう。
ただこちらは自動変換がないので、使うなら`.map_err`との組み合わせになるか。

なお、sourceにできるのは`std::error::Error`トレイトを持っているエラーだけである。

最近は`#[backtrace]`というのもあるそうだ。
使ったことがないので省略。

### その4: なにもなし

付加する情報は無く、エラーの方とメッセージだけ。
例もいるまい。

## どう使うか

### 自分がエラーを返す発端である

何かの関数の結果がエラーだったから自分もエラーを返すというわけではなく、自分で判断してエラーを返すパターン。  
データ長が期待したよりも短いとか、時間がかかりすぎたとか、いろいろあるだろう。

「その2」のタプル型か「その4」の名前だけがよい。
付加する情報がなければ「その4」、あれば「その2」か。  
付加する情報が多いなら「その3」の構造体にするのもよいだろう。

### 元のエラーがある場合

何かの関数を呼んでエラーになったので、そのエラー情報も含めて返したい。

付加する情報がなければ「その1」の`#[from]`、あるなら「その3」の構造体。

`#[from]`は構造体型のシュガーシンタックスというかそういうやつよね。
構造体にしておけばすべてカバーできるので、どう省略したいかを考えればよいだけだ。
