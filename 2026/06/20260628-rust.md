## 基本スタイル

基本的な書き方はこの4パターン。

* [thiserror - Rust](https://docs.rs/thiserror/latest/thiserror/)

```toml
[dependencies]
thiserror = "2.0.18"
```

```rust
use thiserror::Error;

#[derive(Error, Debug)]
pub enum DataStoreError {
    // その1
    #[error("data store disconnected")]
    Disconnect(#[from] io::Error),

    // その2
    #[error("the data for key `{0}` is not available")]
    Redaction(String),

    // その3
    #[error("invalid header (expected {expected:?}, found {found:?})")]
    InvalidHeader {
        expected: String,
        found: String,
    },

    // その4
    #[error("unknown data store error")]
    Unknown,
}
```

### その1: fromを使う

`#[from]`を使うと元のエラー情報を含んだままのエラーにできる。  
また、型が一致したら自動的に変換してくれる。

```rust
use std::{error::Error, fs::File, io, result::Result};

use thiserror::Error;

#[derive(Error, Debug)]
pub enum DataStoreError {
    // その1
    #[error("data store disconnected")]
    Disconnect(#[from] io::Error),
}

fn open() -> Result<(), DataStoreError> {
    let _f =  File::open("abc")?;
    Ok(())
}

fn main() {
    open().unwrap_or_else(|e| {
        eprintln!("{}", e);
        if let Some(err) = e.source()  {
            eprintln!("base error: {}", err);
        }
        eprintln!("-----------------------------");
        eprintln!("{:#?}", e);
    });
}
```

実行すると、`"{}"`では`#[error()]`そのものが、`.source()`で`Some()`が返ってきたら元になったエラーが取得できる。

```log
data store disconnected
base error: No such file or directory (os error 2)
-----------------------------
Disconnect(
    Os {
        code: 2,
        kind: NotFound,
        message: "No such file or directory",
    },
)
```

エラーメッセージを`#[transparent]`で受け流すのはあまりよろしくないそうなので、面倒だが`#[error()]`でなにかメッセージを作ったほうが良いだろう。

`#[from]`は便利なのだがエラーメッセージ以外はいじることができなさそうだ。  

### その2: タプル構造体

これはエラーメッセージを補強するデータとしてタプル構造体を使っている、でいいのかな。
タプルの最初が`{0}`、次が`{1}`のように使える。

```rust
use std::{error::Error, result::Result};

#[derive(thiserror::Error, Debug)]
pub enum DataStoreError {
    // その2
    #[error("the data for key `{0}` is not available")]
    Redaction(String),
}

fn open() -> Result<(), DataStoreError> {
    Err(DataStoreError::Redaction(format!("まったくもう")))
}

fn main() {
    open().unwrap_or_else(|e| {
        eprintln!("{}", e);
        if let Some(err) = e.source()  {
            eprintln!("base error: {}", err);
        }
        eprintln!("-----------------------------");
        eprintln!("{:#?}", e);
    });
}
```

```log
the data for key `まったくもう` is not available
-----------------------------
Redaction(
    "まったくもう",
)
```

### その3: 構造体

その2に近いが、こちらは構造体。お好みで？

```rust
use std::{error::Error, result::Result};

#[derive(thiserror::Error, Debug)]
pub enum DataStoreError {
    // // その1
    // #[error("data store disconnected")]
    // Disconnect(#[from] io::Error),

    // // その2
    // #[error("the data for key `{0}` is not available")]
    // Redaction(String),

    // その3
    #[error("invalid header (expected {expected:?}, found {found:?})")]
    InvalidHeader {
        expected: String,
        found: String,
    },

    // // その4
    // #[error("unknown data store error")]
    // Unknown,
}

fn open() -> Result<(), DataStoreError> {
    Err(DataStoreError::InvalidHeader{expected: "こうであってほしい".to_string(), found: "こうだった".to_string()})
}

fn main() {
    open().unwrap_or_else(|e| {
        eprintln!("{}", e);
        if let Some(err) = e.source()  {
            eprintln!("base error: {}", err);
        }
        eprintln!("-----------------------------");
        eprintln!("{:#?}", e);
    });
}
```

```log
invalid header (expected "こうであってほしい", found "こうだった")
-----------------------------
InvalidHeader {
    expected: "こうであってほしい",
    found: "こうだった",
}
```

構造体の場合は`#[source]`をつけるなりフィールド名を`source`にするなりで`#[from]`のように
フィールドの方が`Error`で終わる名前(`Error`のみも)でもそう判定されるそうだ。  
`#[from]`は便利だけど情報がちょっと足りないというときに使うことになるだろう。
ただこちらは自動変換がないので、使うなら`.map_err`との組み合わせになるか。

なお、sourceにできるのは`std::error::Error`トレイトを持っているエラーだけである。

最近は`#[backtrace]`というのもあるそうだ。
使ったことがないので省略するが `RUST_BACKTRACE=1` を有効にしておくとバックトレースがキャプチャされるとのこと。
panicさせなくても得られるというのは便利そうに見えるな。

### その4: なにもなし

付加する情報は無く、エラーの方とメッセージだけ。
例もいるまい。

## どう使うか

### 自分がエラーを返す発端である

何かの関数の結果がエラーだったから自分もエラーを返すというわけではなく、自分で判断してエラーを返すパターン。  
データ長が期待したよりも短いとか、時間がかかりすぎたとか、いろいろあるだろう。

「その2」のタプル型か「その4」の名前だけがよい。
付加する情報がなければ「その4」、あれば「その2」か。  
付加する情報が多いなら「その3」の構造体にするのもよいだろう。

### 元のエラーがある場合

何かの関数を呼んでエラーになったので、そのエラー情報も含めて返したい。

付加する情報がなければ「その1」の`#[from]`、あるなら「その3」の構造体。

### fromに悩む

`#[from]`は構造体型のシュガーシンタックスというかそういうやつだと思う。
構造体にしておけばすべてカバーできるので、どう省略したいかを考えればよいだけだ。

確かに`#[from]`は`.map_err`を書かなくて済むので楽なのだ。
しかし、変換を書かないで済むためそのエラーが不要になっても気づかない。
せっかくコードがきれいになったのに呼び出し元ではそれがわからないので使われもしないエラー対応を考えないといけないかもしれない、というわけだ。
`cargo clippy`などで検出できればよいのだが、どうも静的なツールではわからんらしい。

`#[error(msg)]`でエラーメッセージをきちんと書けばある程度は発生箇所がわかるのだが、複数個所で発生するとやっぱり情報が足りない。
今やっているのは、それぞれに理由を書いてもらうことだ。
なんとなく`reason`という名前にしたが、発生した関数名とかそういうのを書いたりしているのであんまり理由っぽくないな。。。

```rust
#[derive(Error, Debug)]
pub enum BackendError {
    #[error("Electrum client error occurred: reason={reason}, source={source}")]
    Electrum {
        reason: String,
        #[source]
        source: electrum_client::Error,
    },
}
```

`#[from]`を使わないのであれば`.map_err`で変換するしかなく、変換するのであればその場その場でコードを書くしかなく、
それならそこにユニークな文字列を使うことでエラーメッセージから場所がわかるはずだ。

`#[from]`を廃止することに対する懸念事項を聞いてみると、こんなところだった。
まあ、どれもうなづけるところではある。

* `.map_err`でのコード量が増える
* thiserrorのvariantを増やしてしまいがち
* 自動変換ではないのでマッピングミスをするかも

## `__FILE__`や`__LINE__`のような

C言語だとマクロに現在のファイル名を表す`__FILE__`や今の行番号を表す`__LINE__`がある。
Rustにも同じような`file!()`と`line!()`がある。とGeminiは言っていた。実際にあった。

```rust
use std::{error::Error, result::Result};

#[derive(thiserror::Error, Debug)]
pub enum DataStoreError {
    #[error("invalid header (module={module}, {file}:{line})")]
    InvalidHeader {
        module: &'static str,
        file: &'static str,
        line: u32,
    },
}

fn open() -> Result<(), DataStoreError> {
    Err(DataStoreError::InvalidHeader{module: module_path!(), file: file!(), line: line!()})
}

fn main() {
    open().unwrap_or_else(|e| {
        eprintln!("{}", e);
        if let Some(err) = e.source()  {
            eprintln!("base error: {}", err);
        }
    });
}
```

実行するとこうなった。
無駄な改行をいくつか挟んだりしてみたが、ちゃんと反映した行番号になっている。

```log
invalid header (module=hello, src/main.rs:14)
```

このくらい出しておけばログを埋め込まなくてもよいだろう。
`.map_err`でも`.inspect_err`でもよいのだが、とにかくエラーログを埋め込んでいく作業が面倒と感じてしまってね。。。
ライブラリを使う側だったとしても、エラーが何なのかわかってもコードを見たほうが解決が早いということはしばしばあるし、
もういいんじゃないのか、という気持ちである。

ただ`thiserror`の全variantにこれを埋め込むのは面倒なので、そこもGeminiに考えてもらう。
考えてもらったのは良いが、マクロで解決しよう、だった。
私はまだマクロの書き方を勉強してないので、適当に書き換えて動くようにしたため無理があるかもしれん。

* 行番号などの無い、最初に定義したエラー型 `DataStoreError`
* それをフィールドに持ち行番号などの情報を持つ `LocatedError`
* `LocatedError`を作るのを楽にするトレイト `IntoDetail`
* `file!()`などを中で呼んでくれるマクロ

```rust
use std::{error::Error, result::Result};

#[derive(thiserror::Error, Debug)]
#[error("kind={kind}, module={module} ({file}:{line})")]
pub struct LocatedError {
    pub kind: DataStoreError,
    module: &'static str,
    file: &'static str,
    line: u32,
}

#[derive(thiserror::Error, Debug)]
pub enum DataStoreError {
    #[error("invalid header (expected {expected:?}, found {found:?})")]
    InvalidHeader { expected: String, found: String },
}

pub trait IntoDetail<T> {
    fn with_detail(
        self,
        module: &'static str,
        file: &'static str,
        line: u32,
    ) -> Result<T, LocatedError>;
}

impl<T> IntoDetail<T> for Result<T, DataStoreError> {
    fn with_detail(
        self,
        module: &'static str,
        file: &'static str,
        line: u32,
    ) -> Result<T, LocatedError> {
        self.map_err(|kind| LocatedError {
            kind,
            module,
            file,
            line,
        })
    }
}

#[macro_export]
macro_rules! err_detail {
    ($err_variant:expr) => {
        $err_variant.map_err(|kind| LocatedError {
            kind,
            module: module_path!(),
            file: file!(),
            line: line!(),
        })
    };
}

fn open() -> Result<(), LocatedError> {
    err_detail!(Err(DataStoreError::InvalidHeader {
        expected: "こうであってほしい".to_string(),
        found: "こうだった".to_string(),
    }))
}

fn main() {
    open().unwrap_or_else(|e| {
        eprintln!("{}", e);
        if let Some(err) = e.source() {
            eprintln!("base error: {}", err);
        }
    });
}
```

実行

```log
kind=invalid header (expected "こうであってほしい", found "こうだった"), module=hello (src/main.rs:56)
```

## error_stackクレート

「もっと楽したいんですが！」とGeminiに問うと`error_stack`クレートなるものを紹介してきた。

```toml
[dependencies]
error-stack = "0.7.1"
thiserror = "2.0.18"
```

```rust
use error_stack::{Report, ResultExt};

#[derive(thiserror::Error, Debug)]
pub enum DataStoreError {
    #[error("invalid header")]
    InvalidHeader,
}

fn open() -> Result<(), Report<DataStoreError>> {
    Err(DataStoreError::InvalidHeader).change_context(DataStoreError::InvalidHeader)
}

fn main() {
    open().unwrap_or_else(|e| {
        eprintln!("{}", e);
        eprintln!("---------------------");
        eprintln!("{:?}", e);
    });
}
```

実行

```log
invalid header
---------------------
invalid header
├╴at src/main.rs:10:40
│
╰─▶ invalid header
    ╰╴at src/main.rs:10:40
```

便利そうなのだけど、ちょっと扱いが面倒そうな気がした。  
いや、`thiserror`も最初は扱いが面倒と感じたので慣れの問題かもしれない。
