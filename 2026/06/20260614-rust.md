`thiserror`は便利だ。  
`#[from]`でスルーさせておくととりあえずうまいことやってくれる。

便利でよいのだが、最初はコンパイルエラーになるたびに`#[from]`で追加していけばよいとして、
そのあとでコード整理しているうちに使わなくなるvariantも出てくるだろう。  
そういう使わなくなったvarientは静的にわかるのだろうか？

コンパイルが通っている状態でエラーのvarientを適当に1つ追加してみた。  
`cargo build`では何も警告されない。
コードでunusedなものはコンパイラが比較的洗い出してくれるものの`pub`だと判断できないし、これもそういう気がする(調べてない)。

`cargo clippy`もダメだった。  
最後の希望として`udeps`も試した。ダメだった。

```bash
cargo clippy --all-targets --all-features -- -D warnings
cargo +nightly udeps --all-features
```

`.map_err()`で明示的にしない限りは無理かもしれない。  

ChatGPTには`cargo expand`で展開して"impl From<"で検索したら、と言われた。  
`cargo expand`は`cargo install cargo-expand`でインストールするのだが、expandして"impl From"で検索しても何も出てこなかった。
問い詰めると「`cargo expand`だけでは分かりません」とか言い出す。。。お前はいつもそうだ。。。

もう、variantを1つずつコメントアウトしながらビルドして調べるくらいしか思いつかない。

## コメントアウトしながらエラーを数える

`thiserror`をderiveした`enum`の中身を全部コメントアウトしてビルドし、エラーの数を数える。

```bash
cargo build 2>&1 | grep -c "error"
```

1つコメントアウトを戻して、またビルドしてエラーの数を数える。
エラーの数が減れば意味があるのでそのvariantは残す。
減らなかったらコメントアウトする。

これを全部繰り返せば最後まで終わったときに必要なvariantだけが残っているはずだ。
ただ、エラーの数が減らなかったとしても全部終わったときにまだエラーが残っていて、それを戻すとエラーがゼロになることがあった。
別のエラーがあって検出できなかったとかだろうか？

この方法は確実ではありそうなのだが、手間がかかるし、このAI時代に人間がやることなんだろうかという疑問が出てくる。
でも、これだけのことをAIレビューしてもらうのもトークンがもったいないという貧乏性が出てしまう。  
かといって、Rustのファイルから`thiserror`関係の`enum` variantだけをコメントアウトして1つずつ外してはビルドする、なんていうツールなど作りたくはない。
