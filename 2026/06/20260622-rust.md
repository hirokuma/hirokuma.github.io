## wincode

bがwになっただけだが、なんのwかはわからんかった。

* [anza-xyz/wincode: Fast, bincode‑compatible serializer/deserializer focused on in‑place initialization and direct memory writes.](https://github.com/anza-xyz/wincode)

軽く動かす。

```toml
hex = "0.4.3"
wincode = { version = "0.5.5", features = ["derive"] }
```

```rust
use hex;
use wincode;

#[derive(wincode::SchemaWrite, wincode::SchemaRead)]
struct MyStruct {
    data: Vec<u8>,
    win: bool,
    version: u32,
}

fn main() {
    let val = MyStruct { data: vec![1,2,3], win: true, version: 0x12345678 };
    let enc = wincode::serialize(&val).unwrap();
    println!("enc: {}", hex::encode(enc));
}
```

実行するとこうなった。

```shell
enc: 03000000000000000102030178563412
```

分解するとこうか。  
最初のデータは8バイトあるから、`Vec<u8>`のデータ長を表しているのか。

```txt
???      0300000000000000
data:    010203
win:     01
version: 78563412
```

気になるのは、`version`がリトルエンディアンなことだ。  
これは元のbincodeもそうで、何もしなければリトルエンディアンという仕様になっているとのこと。
[Zero-copy deserialization](https://docs.rs/wincode/0.5.5/wincode/#zero-copy-deserialization)は、複数バイトの型についてリトルエンディアンがターゲットならZero Copyで使いますよといってるのだろう。
これはメモリのアラインメントを考慮して配置しているということだろうか。64bit系のCPUってアラインメントがどれも同じなのかな？  
Arm-Mなんかは奇数アドレスから複数バイト読み出そうとするとたしか例外を起こしてたと思う。

気になるところはあるが、まあきっとちゃんとやってるだろう。
とは思うが確認したい気はするね。

### 参照でも良い

なんとなく実体を渡さないとダメかと思っていたのだが参照でもよかった。

```rust
use hex;
use wincode;

#[derive(wincode::SchemaWrite, wincode::SchemaRead)]
struct MyStruct {
    version: u32,
    data1: Vec<u8>,
    data2: Vec<u8>,
}

#[derive(wincode::SchemaWrite, wincode::SchemaRead)]
struct MyStruct2<'a> {
    version: u32,
    data1: &'a [u8],
    data2: &'a [u8],
}

fn main() {
    let val = MyStruct { version: 0x12345678, data1: vec![1,2,3], data2: vec![4,5,6], };
    let enc = wincode::serialize(&val).unwrap();
    println!("enc: {}", hex::encode(enc));

    let data1 = vec![1,2,3];
    let data2 = vec![4,5,6];
    let val = MyStruct2 { version: 0x12345678, data1: &data1, data2: &data2 };
    let enc = wincode::serialize(&val).unwrap();
    println!("enc: {}", hex::encode(enc));
}
```

```log
enc: 7856341203000000000000000102030300000000000000040506
enc: 7856341203000000000000000102030300000000000000040506
```
