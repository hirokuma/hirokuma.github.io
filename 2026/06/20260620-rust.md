## BIP-38ではない

BIPには秘密鍵をパスフレーズで保護する[BIP-38](https://blog.hirokuma.work/2025/02/20250212-btc.html#bip-38)はある。
しかしこれは単独の秘密鍵についてで、Base58Checkってのが出てくるからWIFなのかな？  
ともかくHDウォレットの拡張秘密鍵についてではない。

どうも拡張秘密鍵についてのBIPはなさそうで、おのおののウォレットでやってるんじゃなかろうかということだった。

## ChaCha20-Poly1305

AIに質問するといくつか方式を挙げてくれた。
ChaCha20-Poly1305はLightning Networkの[BOLT#8](https://github.com/lightning/bolts/blob/master/08-transport.md)でも使っていたので、界隈での筋は悪くなかろうと思い選択した。

ChaCha20とPoly1305を一緒に使う方式で、暗号化についてはストリーム暗号のChaCha20、MACっぽいところはPoly1305を使う。
というほど詳しくない。暗号の仕方がサルサっぽい？というSalsa20から派生してチャチャという名前が付いたとかなんとか。  
AEAD、認証付き暗号と呼ばれる方式だそうで改ざん検知ができるそうだ。なのでMACとは違うのかもしれん。

ChaCha20-Poly1305の注意点はnonceの再利用だそうだ。  
意図的に再利用しなかったとしても標準だと12バイト程度なのでランダム性に注意がいる。

### エンコード

#### Salt

16バイトを`OsRng.fill_bytes()`で埋めている。  
`OsRng`は`bdk_wallet::bitcoin::key::rand::rngs::OsRng`で[rand_core v0.6.4](https://docs.rs/rand_core/0.6.4/rand_core/struct.OsRng.html)だ。
OS由来のRNGを使うようなのでLinuxなら`/dev/urandom`などだろうか。  
まあBDKも使っているようなクレートならよかろう。

そもそもsaltってなんだ。塩を振りかけるようなイメージだったから名前が付いてたような？  
私のフォローしておくと、当時は調べて覚えていたのだよ。忘れたからしょうがないのだ。

* [ソルト (暗号) - Wikipedia](https://ja.wikipedia.org/wiki/%E3%82%BD%E3%83%AB%E3%83%88_%28%E6%9A%97%E5%8F%B7%29)

パスワードを保存せずそのハッシュ値だけ保存しておき、チェックするときはハッシュ値で行うようにする。
そうしておくとハッシュ値が漏れたとしても元のパスワードがわからないとチェックに失敗する。  
が、辞書的なものから総当たりのようにローカルで試し続けることができるので違うデータも付け加える。
パスワードだとだいたい文字列になってしまうが、Saltはそういうのを気にしなくてよい。ユーザも覚えなくてよい。
勝手にハッシュ計算する側が付け加えるだけだ。

なので、パスワードが漏洩するならSaltも漏洩しやすいとは思う。
ただ総当たりの範囲が広くなるので漏洩してから対処するまでの時間は稼ぎやすくなるといったところか。

[MIN_SALT_LEN](https://docs.rs/argon2/latest/argon2/constant.MIN_SALT_LEN.html)(`8`) < len(salt) < [MAX_SALT_LEN](https://docs.rs/argon2/latest/argon2/constant.MAX_SALT_LEN.html)(`0xFFFFFFFF`)

幅が広い。
argon2が使っているバージョンの`password_hash`だと[RECOMMENDED_LENGTH](https://docs.rs/password-hash/0.5.0/password_hash/struct.Salt.html#associatedconstant.RECOMMENDED_LENGTH)は`16`である。  
最新バージョンのargon2だと定義がなくなっているのだが、そもそもargon2を使っていてよいのだろうか？  
ChatGPTに聞くと、ChaCha20-Poly1305を使うならRustCryptoでよかろうということだった。
そして[The RustCrypto Book](https://rustcrypto.org/)の[Password Hashing](https://rustcrypto.org/key-derivation/hashing-password.html#password-hashing)でもargon2を使っているので、そこは大丈夫だと思おう。

ついでにサイズについて聞くと、saltは16バイト、keyは32バイト、nonceは24バイトあればいいんじゃないのということだった。
また、使ってよいならXChaCha20-Poly1305を使おうというアドバイスをもらった。  
Xありとなしの違いはnonceの長さっぽい。↑の24バイトというのはXChaCha20-Poly1305の方だ。
`24`という値を持つ定義はなさそうだった(aeadクレートにはあるらしいがそのために`use`するほどでもなかろう)。

[XChaCha20Poly1305](https://docs.rs/chacha20poly1305/latest/chacha20poly1305/type.XChaCha20Poly1305.html)

#### Argon2

Argon2自体は鍵導出関数の1つである。3種類あって推奨されるのはArgon2idという方式だそうだ。

* [Argon2 - Wikipedia](https://ja.wikipedia.org/wiki/Argon2)

newするときにパラメータを指定しているが、wikipediaに載っていたパラメータ「実行時間、使用できるメモリ量、並列処理の数」のデフォルトが引数3番目の`argon2::Params::default()`に入っている。

```rust
    let argon2 = Argon2::new(
        argon2::Algorithm::Argon2id,
        argon2::Version::V0x13,
        argon2::Params::default(),
    );
```

これがAIレビューに掛けると「明示的に指定したほうが良い」とどれもが言ってくる。
[実装](https://docs.rs/argon2/0.5.3/src/argon2/params.rs.html#85-98)は`m_cost=19 * 1024, t_cost=2, p_cost=1`だ。  
今使っているバージョンは0.5.3で、[0.5.0](https://github.com/RustCrypto/password-hashes/blob/argon2-v0.5.3/argon2/CHANGELOG.md#050-2023-03-04)から[OWASPの推奨値](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html#argon2id)にしたといっているので、
このままにしておくのが無難なんじゃなかろうか。  
バージョンアップによって適切でない値にされる可能性もあるだろうけど、自分で指定するほうが怖い。

そのあと`hash_password_into()`でユーザからのパスワード文字列とさっき作ったsaltをハッシュにして`derived_key`に突っ込み、`ChaChaPoly1305`のインスタンスにしている。

#### Nonce

ChaCha20-Poly1305でさんざん注意されるnonceだ。  
saltと同じく`OsRng.fill_bytes()`を使っているのでよしとしよう。

#### encrypt

先程作った`ChaChaPoly1305`のインスタンスで暗号化する。  
暗号化したいデータは、実際はDescriptor形式の拡張秘密鍵で、そんなに大きくない文字列だ。

`encrypt()`にはそのまま暗号化したいデータだけを与えることもできるし、[Payload構造体](https://docs.rs/aead/0.5.2/aead/struct.Payload.html)にしてadditional associated data(AAD)を付けることもできる。

```rust
    let payload = Payload {
        msg: data.as_bytes(),
        aad: AAD,
    };
```

AADはあった方が良さそうな気がするけどどうなんだろうか？

* [ChatGPT](https://chatgpt.com/share/6a366cb5-9e68-83ee-b11b-5bf3dea915fb)

「暗号化していないメタデータも、暗号文と一体として改ざん検知したい」ということだと、ユーザー名とかになるんだろうか？  
ウォレットの秘密鍵だから改ざんされていて困るかというと、もちろん困りはするのだが改ざんされていた時点でどうしようもない。
もしそれがウォレットとして有効だった場合は開けるし、無効だったら開けないだけだ。
有効だった場合、Bitcoinが入っていれば送金できるし、入っていなければ送金できない。  
あー、自分のウォレットと思って受金用アドレスを他の人に送ったけれども、実は悪意ある人のウォレットから作ったアドレスになるので受金した瞬間に別のアドレスに送られてしまうという可能性はある。  

それだったら改ざんは検出したいが、BDKのウォレットファイルには拡張公開鍵があるので不一致なのがわかるんじゃなかろうか。  
一度作ったBDKウォレットファイルはそのままにして、拡張秘密鍵を別で作って与えてみたところやはりエラーになった。

```log
ERROR sample: wallet: Descriptor mismatch for External keychain: loaded tr([41498c44/86'/1'/0']tpubDCmRJ.....
```

ファイルにバージョン情報を持たせて、バージョンをAADにするのはありかもしれない。

#### zeroize

encryptするのに使った`derived_key`は鍵データなので[zeroise](https://docs.rs/zeroize/latest/zeroize/)で消している。
これもRustCryptoの仲間だ。
中身は全然見ていないが[リポジトリ](https://github.com/RustCrypto/utils/tree/2afc28d5bfc4538296d7b21394892e4c46b612a6/zeroize/src)にCPUの名前があるので、
なるべくatomicな方法でクリアしているのだろう。

ゼロでクリアするのはそれはそれでわかりやすいのでランダム値が良いという話も聞いたことがあるんだがどうなんだろうね。

#### ファイル保存

順番にファイルに書き込んでいるだけだ。  
書き込みに失敗したときのために、別のファイル名で保存してからリネームするのが良いというアドバイスを受けた。  
新規作成ならともかく、上書きでエラーを起こしてどちらのファイルも消えましたというのは最悪だ。  
上書きなんてことは今のところはしないのだけどね。

APIとして、`open(create or read)`とするのか`create(), open()`とするのかは悩ましいところだ。
また、なければ新規作成というやり方もある。
今回は`create(), open()`と別にするタイプで、ウォレットファイルか鍵ファイルかのどちらかでも存在すれば新規作成はできないようにしている。
Bitcoinウォレットは新規のときとそうでないときの手間が違うことが多いので、それにならったというところか。
といいつつも秘密鍵は勝手に保存してユーザにニモニックを見せるわけでもないので、ハードウェアウォレットなどに残したり別のウォレットに持っていくことはできない。
エクスポートできるようにしておけばよいと思いそうだが、Bitcoinのウォレットを使い慣れているとニモニックを表示しないのは心配されるだろう。
